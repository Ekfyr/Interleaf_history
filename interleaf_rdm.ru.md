# Подключение к приложению реляционной базы данных из Interleaf 5 с использованием Lisp

**Чарльз Дейл**\
*Сотрудник технического персонала, Interleaf, Inc.*

----

Программное обеспечение для настольной издательской деятельности **Interleaf 5 (I5)** предоставляет мощную среду кастомизации на основе **Lisp**. Небольшая команда разработчиков, включая автора, использовала эту среду для создания тесно интегрированного интерфейса пользователя (**UI**) для **Relational Document Manager (RDM)** от Interleaf. В этой статье описывается часть программы, реализующей соединение между **I5 Lisp** и **API RDM**.

**RDM** — это система управления документами, поддерживающая крупные проекты по документации, предоставляя три основных компонента функциональности:

1. **Библиотечные услуги**: хранение и извлечение документов, а также отслеживание нескольких черновиков и редакций.
2. **Управление рабочими процессами**: маршрутизация объектов документов и система уведомлений о новых задачах.
3. **Динамический контроль доступа**, основанный на этапе документа в рабочем процессе.

Пользователи создают в **RDM** модель своего проекта по документации, включая структуру различных создаваемых книг и организационную структуру группы документации (авторы, редакторы, рецензенты и т.д.). Объекты **RDM** определяются для представления отдельных документов или иерархий документов (книг). Каждому объекту **RDM** назначается **тип объекта**, который определяет его цикл разработки или маршрутизацию. Маршрутизация — это последовательность этапов, через которые должен пройти объект **RDM** в течение своего цикла разработки. Каждый этап маршрутизации связан с одним или несколькими пользователями **RDM**. Пользователи и группы определяются в организационной структуре модели **RDM**. Объекты **RDM** хранятся в (обычно защищённых) каталогах, называемых **хранилищами (vaults)**.

Все эти сущности — объекты, типы объектов, пользователи, группы и хранилища — хранятся в таблицах **реляционной базы данных**. Основное приложение **RDM** — это программа на **C**, которая включает в себя встроенные операторы **SQL** для доступа и модификации таблиц базы данных. Это приложение не имеет интерфейса пользователя, но предоставляет **API**, к которому можно подключать различные интерфейсы. Мы реализовали три интерфейса: один с использованием переносимого набора инструментов для графического интерфейса **JAM**, работающего под **X** и **Windows**, один с использованием **HyperCard**, и один с использованием **интерфейса рабочего стола I5**.

На дизайн **API RDM** сильно повлиял цикл **read-eval-print** в **Lisp**. Это влияние очевидно в том, что интерпретатор команд **RDM** включает ограниченные модули **READ** и **PRINT** из **Lisp**, способные обрабатывать атомы (без префиксов пакетов), строки (со слешами), числа и вложенные структуры списков. Как будет видно далее, этого достаточно для передачи всего диапазона запросов и ответов **RDM**. С **READ** и **PRINT** в качестве общего знаменателя, связь между интерпретатором команд и **I5** устанавливается с использованием **потоков символов**. В первый раз, когда пользователь вызывает **RDM**, **I5** автоматически запускает интерпретатор команд как отдельный процесс и подключает его стандартный ввод и стандартный вывод к потокам **I5 Lisp**. Затем **I5** выполняет вызовы **RDM**, используя **print** в потоке ввода интерпретатора команд, и получает возвращаемые данные, используя **read** в потоке вывода.

Ввод в интерпретатор команд имеет вид `(запрос список-данных)`. Некоторые запросы не требуют ввода и, следовательно, не нуждаются в `список-данных`. Каждый запрос соответствует функции на **C**, которая принимает список данных в качестве аргумента, интерпретирует различные элементы списка данных и вызывает соответствующую функцию интерфейса базы данных с аргументами, извлечёнными из списка данных. Она также отвечает за сигнализацию об ошибках, вызванных неправильным количеством аргументов или некорректными типами аргументов.

Значения возвращаются из интерпретатора команд в виде `(ошибка список-данных)`. Если запрос вызвал ошибку, `ошибка` — это строка, описывающая проблему; в противном случае это `nil`. Если запрос был запросом на выборку, выбранные данные возвращаются в `список-данных`. Для обновлений `список-данных` — это `nil`. Формат `список-данных` зависит от конкретного запроса, но обычно это список списков, по одному на каждую строку из оператора **SELECT** в **SQL**. Каждый запрос на выборку определяет столбцы или поля, возвращаемые в этих подсписках. Каждое поле — это строка или, в некоторых случаях, список. В примере 1 показан типичный вызов API и возвращаемые значения.

Пример 1: Ввод и вывод данных API

```lisp
Ввод:    (get-user-detail ("CTD"))
Вывод:   (nil
          (("CTD" "USR" "" "CHARLES DALE" "ADMIN" "System Administrator" "SYSTEM"
            "SYSTEM TOP LEVEL" "" "/u/monteverdi/ctd/desktop" "ctd" "")))
```

В этом примере запрос `get-user-detail` принимает один аргумент — `"CTD"`, идентификатор пользователя. Строка, которую он возвращает, содержит 12 столбцов. В этом случае возвращена только одна строка.

Чтобы упростить работу со списками, возвращаемыми из интерпретатора команд, мы используем макрос, который создаёт `defstruct` для определения структур списков и предоставления функций доступа. Структура, возвращаемая в Примере 1, может быть определена следующим образом:

Пример 2: Структура данных API пользователя

```lisp
(define-api-structure user
  name type password full-name default-function function-description parent-group
  group-description last-message work-area mail-address group-vault)
```

Хотя вышеописанный механизм работает, написание большой программы с его использованием было бы очень громоздким. Есть две основные проблемы. Во-первых, каждый раз, когда вы делаете запрос **RDM**, вам нужно проверять значение ошибки и обрабатывать её, если оно не `nil`. Для функций, которые выполняют несколько запросов **RDM**, значительная часть кода должна быть посвящена обработке ошибок API. Это не только ухудшает читаемость кода, но и делает его очень повторяющимся и подверженным ошибкам. Другая проблема заключается в том, что структура аргументов запросов API очень жёсткая, так как она поддерживает только обязательные аргументы. Для вызовов с большим количеством аргументов, таких как поиск объектов, это может быть сложно для программирования и поддержки.

Чтобы сделать доступ к API более похожим на **Lisp**, мы определили **слой API на Lisp**, который посредничает между интерпретатором команд и остальной частью интерфейса пользователя **I5**. Для каждого запроса API мы определяем функцию на **Lisp**, которая предоставляет интерфейс к нему. Функция API на **Lisp** может использовать необязательные и ключевые аргументы и предоставлять значения по умолчанию. Она также заботится о случаях, когда возвращаются ошибки, сигнализируя об **ошибке RDM**.

На рисунке 2 показано, как Lisp API вписывается в общую архитектуру RDM.

![Рисунок 2: Архитектура процесса RDM с использованием Lisp API](/img/Figure_2_RDM_Process_Architecture_with_Lisp_API.png)

Для упрощения программирования с функциями API на **Lisp** предоставлен макрос `handling-api-errors`. API на **Lisp** может вызываться несколько раз в рамках одной формы `handling-api-errors`. По умолчанию каждый вызов API выполняется как отдельная транзакция, но вызовы могут быть сгруппированы в одну транзакцию с использованием вызовов API `open-transaction`, `commit` и `rollback`. Макрос `handling-api-errors` перехватывает условие `rdm-api-error` и выполняет восстановление после ошибки, при необходимости вызывая `rollback`. Он также запускает процесс интерпретатора команд, если он ещё не запущен. Внутри `handling-api-errors` программы используют функции API на **Lisp** как любые другие функции **Lisp**. По сравнению с примером 1, это проще в вызове, а возвращаемые значения проще в использовании.

Пример 3: Ввод и вывод с помощью Lisp API 

```lisp
Ввод:     (get-user-detail "CTD")
Вывод:    (("CTD" "USR" "" "CHARLES DALE" "ADMIN" "System Administrator" "SYSTEM"
            "SYSTEM TOP LEVEL" "" "/u/monteverdi/ctd/desktop" "ctd" ""))
```

В Примере 3, если `get-user-detail` сталкивается с ошибкой, он сигнализирует об ошибке типа `rdm-api-error`, а не возвращает какой-либо вывод. В примере 4 показана простая программа, написанная с использованием Lisp API


Пример 4: Простая программа

```lisp
(defun list-all-users ()
  (handling-api-errors ()
    (terpri)
    (dolist (user (get-user-detail))
      (format t "~12A ~A~4"
        (user-name user)
        (user-full-name user)))))
```

RDM API определяет более 100 запросов, но он разработан таким образом, чтобы быть адаптированным к конкретным приложениям. Чтобы обеспечить максимальную гибкость, у нас есть запрос под названием `execute-sql-statement`, который принимает оператор **SQL** (для защиты целостности базы данных обычно разрешены только операторы **SELECT**) и возвращает полученные строки в подсписках, как описано выше. С помощью этого запроса пользователи могут определять свои собственные новые запросы, структуры и вызовы API. Этот тип интерфейса можно расширить, чтобы предоставить доступ к любой базе данных, предоставляющей интерфейс **SQL**.

В примере 5 показана программа, использующая запрос `execute-sql-statement`. Сначала она определяет новую структуру API, которая отражает список возвращаемых полей. Затем она определяет функцию Lisp API, которая использует функцию `call-api`. Функция `call-api` предназначена для форматирования списка данных, вызова интерпретатора команд, проверки и оповещения об ошибках, а также возврата списка данных из API. На практике `define-api-structure` и `defun` функции API Lisp генерируются одним макросом, называемым `define-api-call`, который также генерирует строку документации и, при необходимости, экспортирует символ функции. Последним шагом является определение функции пользовательского интерфейса. Это выглядит точно так же, как функция в примере 4.

Пример 5: Использование API-вызова `execute-sql-statement`

```lisp
(define-api-structure my-object
  name author)

(defun get-my-objects-by-author (author)
  (call-api 'execute-sql-statement
    (format nil "SELECT object_name, object_author FROM object ~
                 WHERE object_author LIKE '~A' ~
                 ORDER BY object_name"
            author)))

(defun list-my-objects (author)
  (handling-api-errors ()
    (terpri)
    (dolist (my-object (get-my-objects-by-author author))
      (print (my-object-name my-object)))))
```

После того как было потрачено много времени и усилий на разработку и отладку этого кода, самой постоянной проблемой оказалось обеспечение того, чтобы функции **read** и **print** в интерпретаторе команд в достаточной степени соответствовали контрактам своих аналогов на Lisp, чтобы избежать безнадежной путаницы между обеими сторонами. Наиболее распространённая ошибка возникает из-за того, что **Lisp** и **SQL** имеют разные кавычки и другие специальные символы, и правильное экранирование в обоих синтаксисах — это сложная задача. В ретроспективе выбор **read** и **print** для этого интерфейса, возможно, не был лучшим решением. Это определенно упростило реализацию стороны **Lisp**, но добавило много сложности в интерпретатор команд и другие интерфейсы, не на **Lisp**.

Ещё одна проблема с текущей реализацией заключается в том, что **API на Lisp** определяется отдельно от **API на C**. Поскольку разные люди поддерживают каждую сторону этого интерфейса, они могут легко выйти из синхронизации. Один из способов исправить это — усовершенствовать макрос `define-api-call`, чтобы он также генерировал функции интерфейса **API на C**.

Использование Lisp-среды в **Interleaf 5** позволило нам создать очень мощный и гибкий пользовательский интерфейс для RDM. Поскольку большинство установок требуют хотя бы некоторой настройки, упрощение этого процесса было одной из главных задач проектирования. Мы также рассчитываем адаптировать инструменты, разработанные для интерфейса RDM, к другим, более универсальным приложениям баз данных.
